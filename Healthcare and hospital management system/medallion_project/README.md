# Healthcare Medallion Architecture -- Reference Implementation

A working build of the pipeline described in *Healthcare & Hospital Management
System -- Medallion Architecture PDD v2*: 5 flat-CSV sources -> Bronze -> Silver
-> Gold, metadata-driven, with a full audit log and HIPAA-aligned controls.

## What's actually running here vs. the target platform

This was built and tested in a sandboxed local environment with no access to
Databricks or Maven Central, so two substitutions were made. Both are called
out in code comments and are 1-line changes to restore the exact target stack:

| PDD target | This build | Where |
|---|---|---|
| Delta Lake tables | Parquet tables (same read/write API shape) | `utils/spark_utils.py` -> `FORMAT = "delta"` |
| Databricks Workflows scheduling | `notebooks/08_run_pipeline.py` orchestrator script | run manually or wire into cron/Airflow |

Everything else -- the metadata-driven config table, all 24 audit-log columns,
SCD Type 2 on patients, SHA-256 PII masking, the 8 Gold KPIs, and the notebook
boundaries -- follows the PDD exactly (sections 5, 6, 7, 9).

## Running it

```bash
pip install pyspark==3.5.1 --break-system-packages   # delta-spark optional, see note above
python3 generate_source_data.py                       # only needed once, or to regenerate sample data
python3 notebooks/08_run_pipeline.py                   # runs all 8 notebooks in the PDD's Section 8.1 order
```

Each notebook is also independently runnable (`python3 notebooks/06_gold_kpis.py`), matching how they'd
be scheduled as separate tasks in a Databricks Workflow.

## What it produced on the sample data

8 KPIs computed end-to-end from Silver into `gold.gold_kpi_summary`:

| KPI | Value | Target | Status |
|---|---|---|---|
| Patient No-Show Rate | 12.4% | < 10% | AMBER |
| Average Length of Stay | 10.24 days | benchmark by dept | GREEN |
| HCAHPS Satisfaction Score | 80.12% | > 85% | AMBER |
| Billing Accuracy Rate | 96.86% | > 98% | AMBER |
| Insurance Claim Approval Rate | 78.21% | > 95% | RED |
| Medication Inventory Turnover | 112.17x/yr | 4-6x/yr | AMBER |
| Operating Cost per Patient | $13,610.68 | below dept budget | AMBER |
| Patient Readmission Rate | 16.53% | < 15% | AMBER |

These are computed from randomly generated sample data with intentionally
injected data-quality issues (nulls, duplicates, bad charge codes, negative
quantities) — the numbers themselves aren't meaningful, but they prove every
KPI formula in Section 9 executes correctly end-to-end and the alerting logic
(GREEN/AMBER/RED) fires as designed. Swap in real source extracts and the same
notebooks run unchanged.

The audit log (`data/audit/audit_log`) captures one row per source per layer per
run, all 24 columns from Section 7.1, and notebook 07 turns it into a
markdown report at `reports/audit_report.md`.

## HIPAA controls implemented (Section 11)

- **PII masking**: SSN, DOB, phone SHA-256-hashed in Silver (`silver_patients`); raw values dropped, not just hidden.
- **Audit log**: immutable, append-only Delta/Parquet table, every pipeline run logged.
- **No PHI in Gold**: `gold_kpi_summary` and department tables contain only aggregated counts/rates, no patient-level identifiers.
- **Data classification**: carried through the metadata config table and into every audit-log row (PHI/PII/INTERNAL).
- Retention policy, encryption, RBAC, and Unity Catalog column-level security are platform/deployment
  concerns (ADLS Gen2 + Unity Catalog config) and aren't simulated here — they need to be applied when
  this is deployed onto the actual Azure Databricks workspace.

## Known gaps vs. a production build

- No `MERGE INTO` — SCD Type 2 currently just marks every load as a fresh "current" version
  (overwrite mode). A real incremental SCD2 merge (compare against existing `_is_current=true` rows,
  close out changed ones, insert new versions) needs Delta Lake's `MERGE INTO`, which requires the
  actual Delta JAR on Databricks.
- Great Expectations (listed as optional in Section 3.3) isn't wired in; DQ rules here are hand-rolled
  PySpark rather than a GE suite.
- No Databricks Workflows/Unity Catalog config, since those don't exist outside an actual workspace.
